CORE FLOW

記憶 × 模型 × Agent 三角關係

以 BQ Agent 查詢「PurchaseHistory 的 type 欄位代表什麼？」為例

🤖 BQ Agent 接收問題
SOUL.md 決定語氣 • AGENTS.md 決定「先查記憶再查 API」規則
🗃 memory_recall("PurchaseHistory type 欄位")
Qwen3 向量化問題 → LanceDB hybrid 搜尋 → 命中已存 Schema 記憶
🧠 Claude Opus 4.6 推理
結合記憶結果 + 零幻覺原則 → 生成精確回答
type=0:Purchase, type=1:Reverse, type=2:Redeem, type=3:Free, type=4:SpecialComps
💾 memory_store（如有新發現）
新的 Schema 知識 → Qwen3 向量化 → 存入 LanceDB → 下次可直接 recall
LIFECYCLE

BQ Agent Session 完整生命週期

① 使用者在 Telegram @bqbot 發送資料查詢
② Gateway 路由至 BQ Agent → 載入 SOUL + AGENTS + TOOLS + MEMORY.md + 當日日誌
③ memory_recall（Qwen3 → LanceDB hybrid）查相關 Schema / 規則
④ Opus 4.6 推理 → 撰寫 SQL → dry-run 估算成本 → 請求用戶確認
⑤ 用戶確認 → bq-api query 執行 → 結果 + CSV + 圖表 → memory_store
⑥ Telegram 回傳文字摘要 + CSV 檔 + 圖表 PNG → 更新當日日誌

GUARDRAILS

零幻覺原則 & 安全邊界

🚫 零幻覺原則 — 每個欄位名都必須有來源
表名、欄位、數值只能來自以下五個來源，禁止猜測、假設、引用「常見做法」
① memory_recall
② bq-api get-table
③ mssql describe
④ Schema .md 檔案
⑤ 用戶提供
🔒 Read-Only 強制
Wrapper 正則攔截 INSERT、UPDATE、DELETE、DROP、ALTER 等全部 DML/DDL
💰 成本確認制
任何 billable query 都要用戶確認，即使 NT$0.01，dry-run 預估 ≤ NT$60
🛡 職責邊界
只做資料分析。連線、TLS、驅動等基礎設施問題 → 告知用戶聯繫工程師
